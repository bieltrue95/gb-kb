❓Perguntas Clássicas de Entrevista
Por que Strangler Fig em vez de reescrever tudo do zero?▾
Big bang rewrite tem ~70% de taxa de falha. Strangler Fig entrega valor incrementalmente, permite rollback a qualquer momento e mantém o sistema em produção durante toda a migração. O proxy (ex: YARP) roteia tráfego e você migra rota por rota.
Como o hash garante que o mesmo usuário sempre vai para o mesmo bucket de feature flag?▾
SHA256 é determinístico — o mesmo input sempre produz o mesmo output. O fluxo é: userId → SHA256 → primeiros 4 bytes → uint → % 100. O resultado é estável para o mesmo GUID, garantindo experiência consistente sem armazenar estado.
Por que usar jitter no exponential backoff?▾
Sem jitter, todos os clientes que falharam ao mesmo tempo esperam o mesmo delay e tentam novamente ao mesmo tempo — thundering herd. Jitter randomiza o delay dentro de um intervalo, distribuindo a carga no tempo e evitando picos no servidor.
IEnumerable vs IQueryable — quando usar cada um?▾
IQueryable traduz expressões LINQ para SQL via expression trees — o filtro roda no banco. IEnumerable carrega tudo em memória e filtra no C#. Use IQueryable para filtros que devem rodar no banco; use AsEnumerable() quando precisar de operações que só existem em C#.
Qual a diferença entre Singleton, Scoped e Transient no DI do ASP.NET Core?▾
Singleton: uma instância por aplicação. Scoped: uma instância por request HTTP. Transient: nova instância a cada resolução. Armadilha clássica: injetar um Scoped em um Singleton causa "captive dependency" — o Scoped fica vivo mais tempo que deveria.
O que é o Outbox Pattern e qual problema ele resolve?▾
Resolve o problema de dual write: salvar no banco E publicar no broker de forma atômica. Persista o evento em uma tabela outbox na mesma transação. Um background service lê e publica assincronamente. Garante at-least-once delivery e evita mensagens perdidas em caso de falha.
Qual a diferença entre async void e async Task?▾
async void não pode ser awaited e exceções não são capturáveis — elas vão pro thread pool e podem derrubar o processo. Use apenas em event handlers onde a assinatura exige void. Em todo o resto, retorne Task ou Task<T>.
Por que usar record em vez de class para DTOs?▾
Records têm igualdade por valor, imutabilidade por padrão com init-only properties, with-expressions, ToString automático e deconstruct. São ideais para DTOs, value objects e respostas de API onde você quer dados sem identidade mutável. Classes são melhores quando você precisa de estado mutável e identidade de referência.

Architecture & Design
🏗️ SOLID Principles

Os 5 princípios fundamentais de design orientado a objetos para código limpo, extensível e testável em C#/.NET — compilados por Robert C. Martin (Uncle Bob).

5 princ.C# 12.NET 8

Os 5 Princípios

Letra Princípio Resumo Benefício
S Single Responsibility Uma classe deve ter apenas um motivo para mudar Menos acoplamento, mais coesão
O Open/Closed Aberto para extensão, fechado para modificação Adicionar comportamento sem quebrar existente
L Liskov Substitution Subtipos devem ser substituíveis pelo tipo base Hierarquias corretas, sem surpresas
I Interface Segregation Interfaces específicas são melhores que uma genérica Clientes não dependem do que não usam
D Dependency Inversion Dependa de abstrações, não de implementações Desacoplamento total, testabilidade
Como os princípios se conectam: SRP cria classes pequenas e focadas. OCP garante que você estenda sem modificar. LSP valida que hierarquias de herança fazem sentido. ISP evita interfaces inchadas. DIP conecta tudo via abstrações, habilitando DI containers como o do ASP.NET Core.
Por que SOLID é importante em projetos .NET Enterprise?▾
Em projetos grandes, SOLID reduz o custo de mudança. SRP significa que uma alteração no cálculo de imposto não afeta o envio de email. OCP permite adicionar novos tipos de pagamento sem tocar no processador existente. DIP permite trocar o banco de dados de SQL Server para PostgreSQL sem alterar a camada de domínio. Na prática, SOLID é o que torna Clean Architecture e DDD possíveis.

S — Single Responsibility Principle

Uma classe deve ter um único motivo para mudar. Se uma classe faz cálculo de pedido e envia email, ela muda quando a regra de negócio muda e quando o provider de email muda.

❌ Violação: OrderService tem duas responsabilidades 
public class OrderService
{
    public void CreateOrder(Order order)
    {
        // Responsabilidade 1: persistência
        _db.Orders.Add(order);
        _db.SaveChanges();
        // Responsabilidade 2: notificação — motivo diferente para mudar!
        var smtp = new SmtpClient();
        smtp.Send("confirm@shop.com", order.CustomerEmail, ...);
    }
}
✅ Correto: cada classe tem uma responsabilidade 
public class OrderService(IOrderRepository repo, IEmailService email)
{
    public async Task CreateOrderAsync(Order order, CancellationToken ct)
    {
        await repo.SaveAsync(order, ct);
        await email.SendConfirmationAsync(order, ct);
    }
}
public class EmailService(SmtpClient smtp) : IEmailService
{
    public async Task SendConfirmationAsync(Order o, CancellationToken ct) { ... }
}
Qual a diferença entre SRP e coesão?▾
SRP é um princípio de design; coesão é uma métrica de quão relacionadas são as responsabilidades de um módulo. SRP orienta você a ter alta coesão. Uma classe com alta coesão naturalmente tende a cumprir o SRP. A diferença é que SRP foca em razão para mudar (conceito organizacional), enquanto coesão mede o relacionamento funcional entre membros.

O — Open/Closed Principle

Entidades de software devem ser abertas para extensão mas fechadas para modificação. Adicionar um novo comportamento não deve exigir editar código existente. 

// ❌ Violação: adicionar forma exige editar AreaCalculator
public class AreaCalculator
{
    public double Calculate(object shape)
    {
        if (shape is Circle c) return Math.PI * c.Radius * c.Radius;
        if (shape is Rectangle r) return r.Width * r.Height;
        // para adicionar Triangle tem que editar aqui 😢
        throw new NotSupportedException();
    }
}

// ✅ Interface permite extensão sem modificar calculadora
public interface IShape { double Area(); }

public record Circle(double Radius) : IShape
    { public double Area() => Math.PI * Radius * Radius; }
public record Rectangle(double W, double H) : IShape
    { public double Area() => W * H; }
// Triangle novo — ZERO modificação em código existente
public record Triangle(double B, double H) : IShape
    { public double Area() => B * H / 2; }

public class AreaCalculator
{
    public double TotalArea(IEnumerable<IShape> shapes) => shapes.Sum(s => s.Area());
}
Como OCP se relaciona com padrões GoF?▾
Strategy, Template Method, Decorator e Observer são implementações naturais de OCP: permitem adicionar comportamento sem modificar código existente. Strategy troca algoritmos via interface. Decorator adiciona responsabilidades sem modificar a classe base. Observer permite reação a eventos sem o emissor conhecer os ouvintes.

L — Liskov Substitution Principle

Subtipos devem ser substituíveis pela classe base sem quebrar o comportamento. O cliente que usa a base não deve saber que está lidando com uma subclasse. 

// ❌ Square quebra o contrato de Rectangle
public class Rectangle
{
    public virtual int Width  { get; set; }
    public virtual int Height { get; set; }
    public int Area => Width * Height;
}
public class Square : Rectangle
{
    public override int Width  { set { base.Width = base.Height = value; } }
    public override int Height { set { base.Width = base.Height = value; } }
}
// r.Width=3; r.Height=5; Area esperado=15, com Square=25 → LSP violado
Regra prática: se a subclasse precisa lançar NotImplementedException ou ignorar parâmetros do contrato da base, há violação do LSP.
Dê um exemplo real de violação do LSP em .NET.▾
O exemplo clássico é ReadOnlyCollection<T> que herda de Collection<T> e lança NotSupportedException em Add(). Quem recebe Collection<T> e chama Add() não espera exceção. A solução correta seria uma hierarquia diferente com IReadOnlyCollection<T> separado — que é exatamente o que o .NET fez em versões posteriores.

I — Interface Segregation Principle

Clientes não devem ser forçados a depender de métodos que não usam. Prefira interfaces menores e coesas a uma interface grande. 

// ❌ Interface gorda — Robot precisa implementar Eat e Sleep?
public interface IWorker { void Work(); void Eat(); void Sleep(); }

// ✅ Segregar em interfaces pequenas e coesas
public interface IWorkable  { void Work(); }
public interface IFeedable  { void Eat(); }
public interface ISleepable { void Sleep(); }

public class Human : IWorkable, IFeedable, ISleepable { ... }
public class Robot : IWorkable { public void Work() { ... } }

D — Dependency Inversion Principle

Módulos de alto nível não devem depender de módulos de baixo nível. Ambos devem depender de abstrações. É a base do DI container do ASP.NET Core. 

// ❌ Alto nível acoplado ao baixo nível
public class OrderController
{
    private readonly SqlOrderRepository _repo = new(); // new concreto!
}

// ✅ Dependência de abstração — permite trocar SQL por InMemory em testes
public interface IOrderRepository
{
    Task<Order?> GetByIdAsync(int id, CancellationToken ct);
    Task SaveAsync(Order order, CancellationToken ct);
}
public class OrderController(IOrderRepository repo)
{
    [HttpGet("{id}")]
    public async Task<IActionResult> Get(int id, CancellationToken ct)
        => Ok(await repo.GetByIdAsync(id, ct));
}
DIP é a mesma coisa que Dependency Injection?▾
Não. DIP é o princípio (dependa de abstrações). DI é um padrão de implementação que ajuda a cumprir o DIP: em vez de criar dependências internamente (new Repo()), elas são injetadas de fora. O DI container do ASP.NET Core automatiza a injeção, mas o DIP exige que você projete a abstração (interface) primeiro.

Código Completo — SOLID aplicado

// Exemplo completo: sistema de notificação SOLID
// SRP: cada classe uma responsabilidade
// OCP: novo canal (SMS) sem modificar código existente
// LSP: EmailNotifier e SmsNotifier substituíveis por INotifier
// ISP: INotifier tem apenas Send, não FormatTemplate
// DIP: OrderService depende de INotifier, não de EmailClient

public interface INotifier
{
    Task SendAsync(string to, string message, CancellationToken ct);
}

public class EmailNotifier(SmtpClient smtp) : INotifier
{
    public async Task SendAsync(string to, string msg, CancellationToken ct)
        => await smtp.SendMailAsync(to, msg, ct);
}

// Novo canal — ZERO modificação nas classes existentes (OCP)
public class SmsNotifier(ITwilioClient twilio) : INotifier
{
    public async Task SendAsync(string to, string msg, CancellationToken ct)
        => await twilio.SendSmsAsync(to, msg, ct);
}

public class OrderService(IOrderRepository repo, INotifier notifier)
{
    public async Task CreateOrderAsync(Order order, CancellationToken ct)
    {
        await repo.SaveAsync(order, ct);
        await notifier.SendAsync(order.CustomerEmail, "Pedido confirmado!", ct);
    }
}

builder.Services.AddScoped<IOrderRepository, SqlOrderRepository>();

Async & Performance
⏳ Async/Await & Tasks

async/await compõe Tasks para que trabalho I/O-bound (HTTP, DB, arquivo, sockets) rode sem bloquear threads. O compilador gera uma state machine que pausa no await e resume quando a Task completa.

TaskConfigureAwaitCancellationTokenValueTask

Como funciona a state machine

Quando você marca um método com async, o compilador transforma o corpo em uma state machine. Cada await é um ponto de suspensão: se a operação I/O ainda não completou, a thread é liberada e o método retorna uma Task. Quando o I/O completa, a continuação é agendada e o método resume de onde parou.

Call async method
↓
Execute sync part
↓
await I/O → yield thread
↓
I/O completes
↓
Resume after await ✓

I/O-bound vs CPU-bound

Tipo Exemplos Abordagem Bloqueia thread?
I/O-bound HTTP, banco, arquivo, socket async/await Não
CPU-bound Cálculo pesado, criptografia, parsing Task.Run (offload para ThreadPool) Sim (na worker thread)

Por que não se deve usar async void?▾
async void não retorna Task, então o chamador não pode awaitar nem observar exceções. Se uma exceção ocorre, ela vai para o SynchronizationContext e geralmente crasha o processo. A única exceção é em event handlers (ex: Button_Click), onde a assinatura é imposta pelo framework. Em qualquer outro caso, retorne Task.
O que acontece se você esquecer de awaitar uma Task?▾
A Task roda em "fire-and-forget" — exceções são silenciosamente engolidas. O compilador emite warning CS4014 para alertar. Sempre awaite ou armazene a Task explicitamente com descarte (_ = DoAsync()).
ConfigureAwait(false) — quando e por quê?▾
Por padrão, await captura o SynchronizationContext e resume no mesmo contexto. ConfigureAwait(false) pula essa captura — a continuação roda em qualquer thread do pool. Use em código de biblioteca para evitar deadlocks e reduzir overhead. Em ASP.NET Core não há UI context, mas em bibliotecas compartilhadas ainda é boa prática.

CancellationToken

Sempre propague CancellationToken em operações longas. Quando o client abortar a requisição HTTP, o ASP.NET Core cancela o token automaticamente. 

// ASP.NET Core injeta CancellationToken automaticamente no parâmetro
[HttpGet]
public async Task<IActionResult> GetProducts(CancellationToken ct)
{
    var products = await _repo.GetAllAsync(ct);
    return Ok(products);
}

// No repositório: propague sempre
public async Task<List<Product>> GetAllAsync(CancellationToken ct)
    => await _ctx.Products
        .AsNoTracking()
        .ToListAsync(ct); // propaga para o EF Core

// Criar token com timeout manual
using var cts = new CancellationTokenSource(TimeSpan.FromSeconds(5));
try
{
    var result = await _http.GetStringAsync(url, cts.Token);
}
catch (OperationCanceledException)
{
    // timeout ou cancelamento externo
}

Sequential vs Parallel

// Sequential: total ~3s (aguarda cada uma)
var a = await GetAAsync(); // 1s
var b = await GetBAsync(); // 1s
var c = await GetCAsync(); // 1s

// Parallel com Task.WhenAll: total ~1s
var taskA = GetAAsync();   // dispara imediatamente
var taskB = GetBAsync();
var taskC = GetCAsync();
await Task.WhenAll(taskA, taskB, taskC); // aguarda todas
var (ra, rb, rc) = (taskA.Result, taskB.Result, taskC.Result);

// WhenAny — pega o resultado da mais rápida
var fastest = await Task.WhenAny(taskA, taskB, taskC);
var result  = await fastest;
Cuidado: Task.Run em controllers ASP.NET apenas troca threads sem ganho — você sai de uma thread de request para uma thread do pool igualmente bloqueada. Use Task.Run apenas para CPU-bound fora do thread de request.

Async Streams — IAsyncEnumerable

Permite iterar sobre streams de dados assíncrono sem carregar tudo na memória.

// Produtor: yield return assíncrono
public async IAsyncEnumerable<Product> StreamProductsAsync(
    [EnumeratorCancellation] CancellationToken ct = default)
{
    await foreach (var batch in _db.Products.AsAsyncEnumerable().WithCancellation(ct))
    {
        yield return batch; // processa um por vez, sem carregar tudo
    }
}

// Consumidor
await foreach (var product in StreamProductsAsync(ct))
{
    await ProcessAsync(product, ct);
}

Código — State Machine gerada pelo compilador

// O que você escreve:
public async Task<string> GetDataAsync(CancellationToken ct)
{
    var json = await _http.GetStringAsync("https://api.example.com", ct);
    return json.ToUpper();
}

// Simplificado do que o compilador gera (struct de state machine):
private struct GetDataAsyncStateMachine : IAsyncStateMachine
{
    public int _state;
    public AsyncTaskMethodBuilder<string> _builder;
    private TaskAwaiter<string> _awaiter;

    public void MoveNext()
    {
        switch (_state)
        {
            case 0: // início — dispara o request
                _awaiter = _http.GetStringAsync(...).GetAwaiter();
                if (!_awaiter.IsCompleted) { _state = 1; return; } // yield!
                goto case 1;
            case 1: // continuação após I/O
                var json = _awaiter.GetResult();
                _builder.SetResult(json.ToUpper());
                break;
        }
    }
}
/Padrões Distribuídos/Outbox Pattern
Padrões Distribuídos
📬 Outbox Pattern + Idempotência

Regra de ouro: o evento e o estado de negócio commitam na mesma transação de banco. Nunca publique pro broker antes do commit.

Outboxidempotentat-least-once

O problema sem Outbox

Cenário: OrderService salva o pedido no banco e depois publica o evento pro broker. E se o broker estiver fora? O pedido existe, o evento não. O InventoryService nunca decrementa o estoque.

Ordem das operações O que pode dar errado
1. Salva no banco ✓   2. Publica no broker ✗ Evento perdido para sempre
1. Publica no broker ✓   2. Salva no banco ✗ Evento sem dado correspondente
Outbox: ambos na mesma transação Atômico — tudo ou nada
❌ Dual Write — não atômico
↓
SaveChanges() ✓
Broker cai 💥
→ evento perdido para sempre

Solução: Outbox Table

// 1. Entidade Outbox
public class OutboxMessage
{
    public Guid     Id          { get; init; } = Guid.NewGuid();
    public string   Type        { get; init; } = default!; // nome do evento
    public string   Payload     { get; init; } = default!; // JSON serializado
    public DateTime OccurredAt  { get; init; } = DateTime.UtcNow;
    public DateTime? ProcessedAt { get; set; }             // null = pendente
    public string?  Error       { get; set; }
}

// 2. Na mesma transação do domínio
await using var tx = await _ctx.Database.BeginTransactionAsync(ct);

_ctx.Orders.Add(order);
_ctx.OutboxMessages.Add(new
{
    Type    = nameof(OrderCreatedEvent),
    Payload = JsonSerializer.Serialize(new OrderCreatedEvent(order.Id, order.CustomerEmail))
});

await _ctx.SaveChangesAsync(ct); // atômico: order + outbox juntos
await tx.CommitAsync(ct);

Relay Worker — publica a outbox

public class OutboxProcessor(IServiceScopeFactory factory, ILogger<OutboxProcessor> logger)
    : BackgroundService
{
    protected override async Task ExecuteAsync(CancellationToken ct)
    {
        while (!ct.IsCancellationRequested)
        {
            await using var scope = factory.CreateAsyncScope();
            var ctx = scope.ServiceProvider.GetRequiredService<AppDbContext>();
            var bus = scope.ServiceProvider.GetRequiredService<IMessageBus>();

            var pending = await ctx.OutboxMessages
                .Where(m => m.ProcessedAt == null && m.Error == null)
                .OrderBy(m => m.OccurredAt)
                .Take(20)
                .ToListAsync(ct);

            foreach (var msg in pending)
            {
                try
                {
                    await bus.PublishAsync(msg.Type, msg.Payload, ct);
                    msg.ProcessedAt = DateTime.UtcNow;
                }
                catch (Exception ex)
                {
                    msg.Error = ex.Message;
                    logger.LogError(ex, "Failed to publish outbox message {Id}", msg.Id);
                }
            }
            await ctx.SaveChangesAsync(ct);
            await Task.Delay(TimeSpan.FromSeconds(1), ct);
        }
    }
}

Consumer Idempotente

At-least-once delivery significa que o mesmo evento pode ser entregue mais de uma vez. O consumer deve ser idempotente — processar duas vezes deve ter o mesmo efeito que processar uma. 

public async Task HandleAsync(OrderCreatedEvent evt, CancellationToken ct)
{
    // Verifica se já processou usando a chave de deduplicação
    var alreadyProcessed = await _ctx.ProcessedMessages
        .AnyAsync(p => p.MessageId == evt.MessageId, ct);

    if (alreadyProcessed)
    {
        _logger.LogInformation("Duplicate message {Id} ignored", evt.MessageId);
        return; // idempotente: ignora duplicata sem erro
    }

    // Processar o evento...
    await _inventoryService.DecrementStockAsync(evt.ProductId, ct);

    // Marcar como processado (na mesma transação)
    _ctx.ProcessedMessages.Add(new ProcessedMessage { MessageId = evt.MessageId });
    await _ctx.SaveChangesAsync(ct);
}
Outbox garante exactly-once delivery?▾
Não — Outbox garante at-least-once. Se o processor publicar mas cair antes de marcar ProcessedAt, vai publicar novamente. Para efeito de exactly-once: combine Outbox (garante que o evento chega) com consumer idempotente (garante que o efeito só ocorre uma vez). Exactly-once verdadeiro exige coordenação transacional entre banco e broker (Two-Phase Commit) — caro e raramente vale a pena.

Padrões Distribuídos
⚡ Resilience com Polly v8

Circuit Breaker, Retry com exponential backoff e jitter via Polly v8 integrado ao Microsoft.Extensions.Http.Resilience.

Polly v8ResiliencePipelinejitter

Exponential Backoff — fórmula: 2ⁿ segundos

A cada falha o delay dobra. Dá tempo pro serviço se recuperar antes de tentar de novo.

Tentativa Delay base (2ⁿ) Com Jitter Total acumulado
1ª falha 2¹ = 1s 1s + ~250ms = 1.25s ~1.25s
2ª falha 2² = 2s 2s + ~100ms = 2.10s ~3.35s
3ª falha 2³ = 4s 4s + ~400ms = 4.40s ~7.75s
Desiste — falha propagada ao caller —
Fórmula: delay = 2^attempt × baseDelay + random(0, maxJitter)

Por que Jitter é essencial

Sem jitter, todos os clientes que falharam simultaneamente esperam exatamente o mesmo delay e tentam novamente ao mesmo tempo — thundering herd. O servidor upstream recebe uma rajada exatamente quando estava se recuperando.

❌ Sem Jitter: thundering herd
↓
Client A: retry em 2s
Client B: retry em 2s
Client C: retry em 2s
→ todos batem no servidor ao mesmo tempo
✅ Com Jitter: distribuído no tempo
↓
Client A: 1.8s
Client B: 2.3s
Client C: 1.5s

Circuit Breaker — os 3 estados

Closed — operação normal
↓ falhas > threshold na janela
Open — fast fail imediato
↓ após BreakDuration
Half-Open — testa 1 request probe
↓ sucesso → Closed | falha → Open
Estado Comportamento Transição
Closed Chamadas passam normalmente → Open quando falhas > threshold
Open Fail fast — rejeita sem chamar downstream → Half-Open após BreakDuration
Half-Open Testa com probe requests → Closed (sucesso) ou Open (falha)
Quando Circuit Breaker é mais importante que Retry?▾
Retry assume que a falha é transitória. Quando o serviço downstream está completamente indisponível, retry com backoff apenas atrasa a falha e desperdiça recursos. Circuit Breaker detecta essa situação e para de tentar imediatamente — fail fast — liberando recursos e dando tempo ao serviço se recuperar. Use ambos em conjunto: retry para falhas transitórias, circuit breaker para outages.

Polly v8 — ResiliencePipeline no .NET 8

// Program.cs — via Microsoft.Extensions.Http.Resilience
builder.Services
    .AddHttpClient<IProductApiClient, ProductApiClient>()
    .AddResilienceHandler("product-pipeline", pipeline =>
    {
        // 1. Timeout por tentativa
        pipeline.AddTimeout(TimeSpan.FromSeconds(3));

        // 2. Retry com exponential backoff + jitter
        pipeline.AddRetry(new HttpRetryStrategyOptions
        {
            MaxRetryAttempts = 3,
            Delay            = TimeSpan.FromSeconds(1),
            BackoffType      = DelayBackoffType.Exponential,
            UseJitter        = true,
            ShouldHandle     = new PredicateBuilder<HttpResponseMessage>()
                .Handle<HttpRequestException>()
                .HandleResult(r => (int)r.StatusCode >= 500)
        });

        // 3. Circuit Breaker
        pipeline.AddCircuitBreaker(new HttpCircuitBreakerStrategyOptions
        {
            FailureRatio      = 0.5,  // abre com 50% de falhas
            SamplingDuration  = TimeSpan.FromSeconds(10),
            MinimumThroughput = 10,
            BreakDuration     = TimeSpan.FromSeconds(30)
        });
    });

// Via ResiliencePipelineBuilder direto (sem HttpClient)
var pipeline = new ResiliencePipelineBuilder()
    .AddRetry(new RetryStrategyOptions
    {
        MaxRetryAttempts = 3,
        UseJitter        = true,
        OnRetry = args =>
        {
            logger.LogWarning("Retry {Attempt} after {Delay}",
                args.AttemptNumber, args.RetryDelay);
            return default;
        }
    })
    .Build();

await pipeline.ExecuteAsync(async ct =>
    await _http.GetStringAsync("https://api.example.com", ct), ct);

Architecture & Design
💉 Dependency Injection

DI fornece dependências de fora (via construtor) em vez de criá-las internamente. Desacopla o código, melhora a testabilidade e torna os lifetimes explícitos.

lifetimesIServiceCollectioncaptive dependency

O que é Dependency Injection?

DI é um padrão onde uma classe recebe suas dependências em vez de criar internamente. O mecanismo mais comum é Constructor Injection: a dependência é passada como parâmetro do construtor. 

public interface IClock { DateTime UtcNow { get; } }
public sealed class SystemClock : IClock
    { public DateTime UtcNow => DateTime.UtcNow; }

public sealed class Greeter(IClock clock)
{
    public string Greet(string name) => $"[{clock.UtcNow:o}] Hello, {name}!";
}
// Greeter depende da ABSTRAÇÃO (IClock), não do tipo concreto.
// Em testes: injete FakeClock. Em produção: injete SystemClock.

Composition Root

O único lugar onde todas as dependências são registradas e conectadas. Em ASP.NET Core, isso é o Program.cs. Nenhuma outra classe deve saber que o container existe. 

// Program.cs — o Composition Root
var builder = WebApplication.CreateBuilder(args);

builder.Services.AddSingleton<IClock, SystemClock>();
builder.Services.AddTransient<Greeter>();
builder.Services.AddScoped<IOrderRepository, SqlOrderRepository>();
builder.Services.AddDbContext<AppDbContext>(opt =>
    opt.UseSqlServer(connectionString));
// Daqui pra frente, nenhuma classe sabe que o container existe

Captive Dependency — bug silencioso

Ocorre quando um serviço de vida longa captura um serviço de vida curta. 

// ❌ Singleton capturando Scoped — DbContext vivo para sempre!
public class CacheService(AppDbContext db) // Singleton capturando Scoped
{
    // _db será o mesmo DbContext em todos os requests → problema de estado
}

// ✅ Singleton que precisa de Scoped: use IServiceScopeFactory
public class CacheService(IServiceScopeFactory factory)
{
    public async Task RefreshAsync(CancellationToken ct)
    {
        await using var scope = factory.CreateAsyncScope();
        var db = scope.ServiceProvider.GetRequiredService<AppDbContext>();
        // db é Scoped dentro deste scope temporário — correto
    }
}
Por que DbContext deve ser Scoped e não Singleton?▾
DbContext mantém estado — tracked entities, transações abertas, change tracker. Se for Singleton, será compartilhado entre requests concorrentes, causando condições de corrida e dados incorretos. Scoped garante um DbContext isolado por request, descartado ao fim do request.

Service Locator — Anti-Pattern

É quando você injeta IServiceProvider e chama GetService<T> manualmente dentro de um serviço. Esconde as dependências reais e dificulta testes. 

// ❌ Service Locator — anti-pattern
public class OrderService(IServiceProvider sp)
{
    public async Task ProcessAsync(Order order)
    {
        var repo = sp.GetService<IOrderRepository>(); // oculta dependências
        var email = sp.GetService<IEmailService>();   // impossível de mockar facilmente
    }
}

// ✅ Constructor Injection — dependências visíveis e mockáveis
public class OrderService(IOrderRepository repo, IEmailService email)
{ ... }
Existe algum caso válido para Service Locator?▾
Sim — em fábricas onde o tipo concreto só é conhecido em runtime (ex: plugin system), ou em middleware onde o lifetime precisa ser controlado manualmente com IServiceScopeFactory. A chave é que o uso deve ser exceção, não regra, e o Service Locator deve ser encapsulado, não espalhado pelo código de domínio.

Quando usar AsNoTracking()?▾
Use em queries de leitura onde você não vai modificar as entidades. O Change Tracker tem custo de memória e CPU para monitorar cada entidade carregada. Em relatórios, listagens e endpoints GET, AsNoTracking() pode reduzir significativamente o tempo de query e uso de memória.
Diferença entre Fluent API e Data Annotations?▾
Data Annotations são atributos no modelo de domínio — acoplam domínio e persistência. Fluent API é configurada em classes IEntityTypeConfiguration separadas — mantém o domínio limpo de referências ao EF Core. Prefira Fluent API em projetos com Clean Architecture ou DDD.

O que é Split Query e quando usar?▾
Ao usar Include em coleções, EF Core gera um JOIN que pode criar produto cartesiano enorme. AsSplitQuery() divide em múltiplas queries menores. Use quando o JOIN produziria muitas linhas duplicadas. Desvantagem: não é atômica — dados podem mudar entre as queries. Ative globalmente com UseQuerySplittingBehavior(QuerySplittingBehavior.SplitQuery).
Como resolver o JsonIgnore problem com navigation properties?▾
Navigation properties com circular reference causam loop de serialização. Solução 1: [JsonIgnore] na navigation property da entidade. Solução 2 (melhor): usar DTOs/projeções e nunca serializar a entidade diretamente. Solução 3: configurar ReferenceHandler.IgnoreCycles no JsonSerializerOptions globalmente no ASP.NET Core.

public string GenerateToken(User user)
{
    var claims = new[]
    {
        new Claim(ClaimTypes.NameIdentifier, user.Id.ToString()),
        new Claim(ClaimTypes.Email, user.Email),
        new Claim(ClaimTypes.Role, user.Role),
        new Claim("tenant_id", user.TenantId.ToString())
    };
    var key   = new SymmetricSecurityKey(Encoding.UTF8.GetBytes(_secret));
    var creds = new SigningCredentials(key, SecurityAlgorithms.HmacSha256);
    var token = new JwtSecurityToken(
        issuer: _issuer, audience: _audience,
        claims: claims,
        expires: DateTime.UtcNow.AddHours(1),
        signingCredentials: creds);
    return new JwtSecurityTokenHandler().WriteToken(token);
}

Como revogar um JWT antes do vencimento?▾
JWT é stateless — sem revogação nativa. Estratégias: 1) Token blocklist: manter lista de JTI revogados no Redis com TTL = tempo restante. 2) Access token curto (15min) + Refresh token longo no banco — ao revogar, exclui o refresh. 3) Versão de token: gravar versão na claim, incrementar no banco ao logout — middleware valida versão a cada request.
O que é PKCE e quando é obrigatório?▾
PKCE (Proof Key for Code Exchange) adiciona um code_verifier (string aleatória) e code_challenge (SHA256 do verifier) ao fluxo Authorization Code. O servidor verifica que quem troca o code pelo token é quem iniciou o fluxo — evita interceptação do authorization code. É obrigatório para SPAs e apps mobile (public clients) e recomendado para todos os fluxos OAuth 2.1.

Cache-Aside Pattern

public class ProductService(IDistributedCache cache, IProductRepository repo)
{
    public async Task<ProductDto?> GetByIdAsync(int id, CancellationToken ct)
    {
        var key    = $"product:{id}";
        var cached = await cache.GetStringAsync(key, ct);

        if (cached is not null)
            return JsonSerializer.Deserialize<ProductDto>(cached);

        var product = await repo.GetByIdAsync(id, ct);
        if (product is null) return null;

        var opts = new DistributedCacheEntryOptions
        {
            AbsoluteExpirationRelativeToNow = TimeSpan.FromMinutes(10),
            SlidingExpiration               = TimeSpan.FromMinutes(2)
        };
        await cache.SetStringAsync(key,
            JsonSerializer.Serialize(product.Adapt<ProductDto>()), opts, ct);

        return product.Adapt<ProductDto>();
    }

    public async Task InvalidateAsync(int id, CancellationToken ct)
        => await cache.RemoveAsync($"product:{id}", ct);
}

LINQ usa execução diferida: a query não roda até você iterar ou chamar um operador terminal. Entender quando o SQL é gerado é crucial para performance em EF Core.

Deferred vs Immediate Execution

Tipo Execução Exemplos
Deferred Ao iterar Where, Select, OrderBy, GroupBy
Immediate Na chamada ToList(), ToArray(), Count(), First(), Sum()

IEnumerable vs IQueryable

// IEnumerable: filtra em memória (C#) — traz TUDO do banco
IEnumerable<Product> products = _ctx.Products; // consulta executa aqui!
var cheap = products.Where(p => p.Price < 50);  // filtra em RAM

// IQueryable: traduz para SQL — filtra no banco
IQueryable<Product> query = _ctx.Products;       // sem hit no banco
var cheap = query
    .Where(p => p.Price < 50)    // adiciona WHERE ao SQL
    .OrderBy(p => p.Name)          // adiciona ORDER BY
    .Take(10)                     // adiciona TOP 10
    .ToListAsync(ct);              // ← AQUI executa: SELECT TOP 10...

Armadilha: closure em loop

// ❌ Captura a variável, não o valor!
var queries = new List<Func<int>>();
for (int i = 0; i < 5; i++)
    queries.Add(() => i); // captura 'i' por referência
queries.ForEach(f => Console.Write(f())); // 55555 — todos pegam i=5

// ✅ Capturar cópia local
for (int i = 0; i < 5; i++)
{
    int copy = i;
    queries.Add(() => copy); // captura 'copy', que não muda
}
// 01234 — correto
O que são expression trees e como o EF Core as usa?▾
IQueryable<T> aceita Expression<Func<T, bool>> em vez de Func<T, bool>. Expression trees são representações em memória do código como dados — o EF Core as inspeciona e traduz para SQL. Um Func seria código compilado opaco; uma Expression é uma árvore que pode ser lida e transformada pelo provedor LINQ.
